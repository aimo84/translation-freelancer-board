Add Error Log

Performing the review

Go to the Error Log tab.
If you have entered your language and the date, these will show on the relevant cells.
For each error identified, proceed as follows:

  1. Enter the location (file name, page number, etc) where the error was found.
  2. Select an error category from the dropdown list. See below for definitions of available error categories.
  3. Select the severity from the dropdown list. See below for definitions of severity levels.
  4. Enter the source term or sentence, the translation used, and your suggested correction.
  5. Please add further comments in English to clarify the reason for your correction. This should be done in a professional tone, with no confrontational or personal comments and in a way so that a non-native speaker would be able to understand.
  6. Please make sure to report only errors, rather than subjective preferences. If you think that a personal choice would be more suitable, please mark this accordingly and do not count it in the final results.
  7. Please change the name of the file to include the language two-letter identifier, as per ISO 639 standard.
NOTE: if your suggestions are desirable but do not highlight an issue, please list them in the Error Log anyway and add a comment to elaborate on your suggestion. No severity will be necessary but it may be useful to list the error category.

Updating the QA Summary tab

Go to the QA Summary tab.
For each error category, the error points score is automatically calculated (1 point for each minor error, 5 points for each major error, maximum amount of points plus one for each critical error).
A PASS is achieved if the total number of error points is below or equivalent to the maximum number of error points allowed.
If the number of total errors is above the allowed quota, the form will automatically show a FAIL score.

NOTE: If the number of error points allowed for a category is exceeded but the overall result is still within the allowed threshold, the formula will still calculate a PASS. The language lead/the PM may override this depending on client requirements and manually default the result to a FAIL.

Error categories:

Category
Examples

Mistranslation
    Source text incorrectly rendered when no authorization has been given to justify alterations.
Accuracy
  • Unauthorized omissions.
  • Unauthorized additions.
  • Headers/Footers incorrectly translated.
  • Text accidentally left in the source language.
  • Project instructions not followed.
  • Query feedback not implemented.
  • Undertranslation/Overtranslation.
Terminology
  • Project or client glossary not followed.
  • OS terminology not followed
  • Incorrect UI references (validated UI mismatch).
  • Industry-standard terminology not followed.
  • Inconsistent use of terminology.
  • Shortcut keys not correctly localized.
Language
  • Grammatical errors
  • Semantics (ambiguous meaning)
  • Syntactical errors
  • Spelling errors
  • Punctuation errors
  • Typos
Style
  • Style conventions not adhered to.
  • Unidiomatic use of target language
  • Target text follows source text too closely.
  • Register/tone not appropriate for target audience.
  • Note: it is often difficult to capture bad style in terms of individual errors. The reviewer may generic log a comment in the Comments space under the QA Summary tab, providing as many examples as possible.
Country
  • Date/time format not adapted to local standards.
  • Currency symbol not adapted to local standard, amounts not converted.
  • Examples not relevant to target culture (examples, Italian students sitting a spelling test in a standard Italian school).
  • Imperial measurements not converted to metric system.
  • Local contact information not implemented although instructions were provided in this respect.
  • Source part numbers not amended with target ones although as per instructions.
Consistency
  • Cross-references
  • TOC/Index
  • Help contents file inconsistent with topic titles.
  • Headers structure
  • Note: For issues with inconsistent terminology, please refer to the Terminology section

Severity levels:

Severity
Examples

Critical
  • Error in particularly visible part of documentation or software
  • Major error repeated in several locations
  • Error causing the application to crash or negatively modifying/misrepresenting functionality of the software
  • Error resulting in potentially offensive statement
Major
  • Error in highly visible part of documentation or software (header, TOC, menu command, help/CBT topic)
  • Error resulting in a misleading statement
  • Minor error repeated in several locations
  • Query answers or QA feedback not applied
  • Generic error to indicate generally inadequate style
Minor
  • Any error of a lesser severity than major
  • Note: repeated minor errors may be counted as one major error

Repeat errors:

If an error occurs three times or more, do not list the separate occurrences. Log only one occurrence and choose Yes in this column.
